设计宣言Design Manifesto

AI 协作系统最大的浪费，不是算力，而是遗忘。
每次对话重新解释上下文、每次任务重新定位模块、每次变更重新梳理约束——
这些摩擦的根源是：能力与业务没有被分开治理，经验没有被系统性沉淀。

CBIM 的核心主张是：让系统在执行中自我成长。
能力轴管 agent 的生命周期，业务轴管知识的生命周期，记忆把执行经验持续提炼为两轴的养料。
项目越复杂、时间越长，CBIM 的复利效应越显著。

The greatest waste in AI collaboration is not compute — it's forgetting.
Re-explaining context every session, re-locating modules every task, re-tracing constraints every change —
the root cause is that capability and business are not governed separately,
and experience is not systematically distilled.

CBIM's core premise: let the system grow while it executes.
The capability axis governs agent lifecycles, the business axis governs knowledge lifecycles,
and memory continuously refines execution experience into nutrients for both.
The more complex the project and the longer it runs, the more significant CBIM's compounding returns.

Capability · Business · Independence · Memory

六大核心原则Six Core Principles

⚔️

能力与业务解耦 Capability-Business Decoupling

Agent（谁来做）与 DNA 模块（做什么、怎么约束）彻底分离。能力层可以招募新 agent 而不破坏业务知识；业务层可以演化架构而不重写 agent 定义。

Agent (who does it) and DNA modules (what to do, how to constrain) are fully decoupled. The capability layer can recruit new agents without disrupting business knowledge; the business layer can evolve architecture without rewriting agent definitions.

Capability-Business Independence
🧠

记忆是一等公民 Memory as First-Class Citizen

执行经验不随会话消失。三层沉淀（short → medium → .dna/）将偶发经验提炼为稳定原则，让每次对话站在所有历史积累的肩膀上。

Execution experience doesn't vanish with the session. Three-tier distillation (short → medium → .dna/) refines ad-hoc experience into stable principles, so every conversation stands on the shoulders of all prior accumulation.

Memory-First
🌱

知识懒式生成 Lazy Knowledge Generation

DNA 文档不是前置必做项，而是在真正需要时由 Architect 按需创建。避免过度文档化，聚焦在实际触碰到的模块边界和约束上。

DNA documents are not prerequisites — Architect creates them on demand when they are actually needed. Avoids over-documentation; focuses on module boundaries and constraints that are truly touched.

Lazy Knowledge Generation
🔄

自治调度循环 Autonomous Dispatch Loop

用户心智负担最低化。Coordinator 自治完成「理解→分解→派发→收结果→分析→再派发」的循环，只在真正需要决策时打断用户。

Minimal cognitive load on the user. The Coordinator autonomously completes "understand → decompose → dispatch → collect → analyze → re-dispatch" and only interrupts when a real decision is required.

Vibe Working
📐

双轴对偶治理 Dual-Axis Symmetric Governance

能力轴（.claude/agents/）与业务轴（.dna/）对称治理、协同裂变。新业务需求催生新 agent，新 agent 承载新业务知识，两轴互相驱动边界扩展。

The capability axis (.claude/agents/) and the business axis (.dna/) govern symmetrically and undergo coupled fission. New business needs spawn new agents, new agents carry new business knowledge, and both axes drive each other's boundary expansion.

Dual-Axis Governance
📈

复利飞轮效应 Compounding Flywheel

每次任务执行都在让系统变得更好——DNA 状态从「无」到「同步」，agent 从「通用」到「专精」，记忆从「碎片」到「原则」。项目规模越大、运行时间越长，优势越显著。

Every execution makes the system better — DNA states evolve from "absent" to "synced," agents from "generic" to "specialized," memory from "fragments" to "principles." Advantages compound with project scale and duration.

Compounding Returns

游戏开发拥有大量细分工种（程序、策划、美术、音频、QA、DevOps）和高度解耦的业务模块。 每个系统都有独立的架构约束和接口契约，工种之间频繁协作又需要严格边界。 这正是 CBIM 双轴治理复利最大化的场景：能力轴管人，业务轴管模块，两轴协同裂变，项目越大越不乱。

Game development has many specialized roles (programming, design, art, audio, QA, DevOps) and highly decoupled business modules. Each system carries its own architectural constraints and interface contracts; roles collaborate frequently yet need strict boundaries. This is exactly where CBIM's dual-axis governance compounds most: the capability axis governs people, the business axis governs modules, the two co-evolve, and the bigger the project, the more it stays in order.

✅ CBIM 价值显著的场景 ✅ Where CBIM Shines

  • 大体量项目：模块数量多、业务复杂度高、需要长期维护 Large projects with many modules, high complexity, long-term maintenance
  • 多子工种协作：不同角色（策划/程序/美术/QA）需要各自专属 agent Multi-role collaboration — design/programming/art/QA each need dedicated agents
  • 高频跨模块变更：改动频繁触及模块边界，需要约束守护 Frequent cross-module changes that constantly touch module boundaries and need guardrails
  • 长期运营项目：积累的记忆和 DNA 随时间形成复利 Long-running products where accumulated memory and DNA compound over time
  • 大型游戏开发：战斗/背包/剧情/AI/渲染等系统并行演化 Large-scale game development — combat / inventory / story / AI / rendering evolve in parallel
  • 中台/平台型系统：多业务线共用底层模块，契约稳定性关键 Platform / middle-office systems — multiple business lines share core modules; contract stability is critical

✗ CBIM 作用有限的场景 ✗ Where CBIM Adds Little

  • 小型一次性项目：建立 DNA + agent 体系的成本高于收益 Small one-off projects — setting up DNA + agents costs more than it saves
  • 单人快速原型：没有多工种协作需求，能力轴空转 Solo rapid prototypes — no multi-role need; the capability axis idles
  • 短生命周期脚本：写完即弃，无需知识沉淀 Short-lived scripts — write-and-discard; no knowledge to sediment
  • 业务逻辑极简：模块少、边界清晰，DNA 治理过重 Minimal business logic — few modules, clear boundaries; DNA governance is overkill
  • 纯探索性 Spike：目的是验证可行性而非构建可维护系统 Pure exploratory spikes — the goal is feasibility, not a maintainable system

判断标准：如果你能一眼数清楚项目的所有模块（少于 10 个），且只有 1-2 个开发角色，CBIM 的治理成本可能超过收益。 当模块数超过 20、工种超过 3 种、或者项目预计持续超过半年，CBIM 的复利效应开始显现。
游戏开发是 CBIM 最自然的宿主——它天然拥有大量细分领域、长期演化需求和多工种协作，完美匹配双轴治理的设计哲学。

Rule of thumb: if you can count every module at a glance (fewer than 10) and have only 1–2 dev roles, CBIM's governance cost may exceed its benefit. Once modules exceed 20, roles exceed 3, or the project is expected to run beyond six months, CBIM's compounding effect starts showing.
Game development is CBIM's most natural host — it inherently brings many specialized domains, long-running evolution, and multi-role collaboration, perfectly matched to dual-axis governance.
