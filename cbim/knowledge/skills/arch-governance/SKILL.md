# Skill: 知识治理与合规（架构师）

> 定期审查 `.aimodule/` 知识体系的健康状态，发现并修复架构违规。

## 触发场景

- 新建/更新/废弃模块后（必须执行快速检查）
- 助手发起定期巡检
- 评审官指出架构问题后

---

## 快速检查（每次模块变更后）

```bash
python cbim/knowledge/modules.py list   # 确认模块树完整
```

检查新建/更新模块：
- [ ] `module.json` 的 `name`（kebab-case）和 `owner` 已填写
- [ ] `architecture.md` 和 `contract.md` 内容不为模板占位符
- [ ] 根模块 `index.md` 已同步更新
- [ ] 新增依赖无循环（A→B→A 类型）

---

## 全量合规巡检

### Step 1 — 扫描模块树

```bash
python cbim/knowledge/modules.py list --root .
```

逐模块检查：

| 检查项 | 合规标准 |
|--------|---------|
| 单一职责 | 模块功能能一句话说清 |
| 单向依赖 | `dependencies` 不存在循环 |
| 知识完整 | `architecture.md` 和 `contract.md` 均有实质内容 |
| 边界清晰 | `architecture.md` 只含内部设计，`contract.md` 只含对外接口 |
| index 同步 | 根模块 `index.md` 与实际模块目录一致 |

### Step 2 — 检查跨模块依赖方向

依赖只能从"易变"流向"稳定"：
- 业务模块 → 基础设施模块 ✅
- 基础设施模块 → 业务模块 ❌

发现反向依赖 → 引入接口层解耦，稳定侧拥有接口定义权（C3）。

### Step 3 — 输出治理报告

```
巡检时间：<date>
模块总数：<N>
发现问题：
  - <模块路径>: <问题描述>
  - ...
已修复：<列表>
待处理（需用户确认）：<列表>
```

### Step 4 — 修复或上报

- 可自主修复（补充文档、更新 index）→ 直接修复，汇报结果
- 涉及接口重构、模块拆分 → 汇报助手，由用户确认后执行
