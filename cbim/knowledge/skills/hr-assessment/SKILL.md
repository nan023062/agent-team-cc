# Skill: Agent 评审（HR）

> 模拟资深 leader review：对 agent 定义进行两个维度的结构化审查：
> 1. **定义合理性** — 职责单一、能力聚焦、格式规范
> 2. **定义与表现一致性** — `.claude/agents/` 文档与实际执行行为是否对齐
>
> 输出等同于 leader review comment：指出问题、判断严重性、给出处置建议。

**铁律**：agent 定义（soul + skills）只含纯能力维度，不含任何项目特定内容。
项目内容（模块名、业务逻辑、具体路径）属于内容层，只存在于模块知识库（`.dna/`）。

## 触发场景

- 一批任务完成后（由助手或用户发起）
- 用户明确指出某 agent 表现不足
- 评审官连续打回同一 agent 的交付物（≥ 2 次）
- 定期巡检（全部 work agent）

---

## 快速检查（单次任务结束后）

仅对本次参与任务的 agent 执行基础格式检查（因子 1–3）：

```bash
python cbim/knowledge/engine/cli.py agents show <name>
```

- [ ] frontmatter 字段完整（`name`、`description`、`model`、`tools`）
- [ ] `description` 一句话说清定位，可供助手派发决策直接使用
- [ ] Skills 表中的所有 `SKILL.md` 路径实际存在

---

## 全量 Agent 评审

### 准备：扫描 Agent 列表

```bash
python cbim/knowledge/engine/cli.py agents list
```

逐个 agent 执行以下五步审查。

---

### 第一步 — 基础格式

| # | 因子 | 合规标准 |
|---|------|---------|
| 1 | frontmatter 完整 | `name`、`description`、`model`、`tools` 均已填写 |
| 2 | description 可用 | 一句话定位，助手读到即可决策是否派发，无歧义 |
| 3 | skills 路径有效 | Skills 表中每个 `SKILL.md` 路径实际存在且可读 |

---

### 第二步 — 职责单一性

| # | 因子 | 检查方法 |
|---|------|---------|
| 4 | 职责单一 | 职责描述能在当前专业方向下一句话说清；若描述含"以及""同时负责"等，视为职责过宽 |
| 5 | 触发场景聚焦 | 触发场景列表中的所有条目指向同一类需求，无跨域兼职 |

---

### 第三步 — 能力聚焦度

| # | 因子 | 检查方法 |
|---|------|---------|
| 6 | skills 方向一致 | 所有 skill 指向同一专业方向，无能力漂移（如一个 agent 同时有业务开发 + 运维 + 审查 skill） |
| 7 | skills 数量合理 | skills ≥ 5 个时触发裂变建议：能力过广，应拆分为多个专精 agent |
| 8 | tools 最小必要 | 授权的 tools 只包含该职责实际需要的，无多余授权 |

---

### 第四步 — 内容纯洁性（铁律）

agent 定义是**可移植能力描述**，不是项目文档。

| # | 因子 | 检查方法 |
|---|------|---------|
| 9 | soul 无项目内容 | `<id>.md` 正文不含：项目名、模块名、业务逻辑、当前任务状态、具体文件路径（cbim/ 框架路径除外） |
| 10 | skills 无项目内容 | 各 `SKILL.md` 只描述操作方法与判断原则，不引用具体项目的模块名或业务规则 |
| 11 | 可移植性自检 | 将该 agent 定义放入另一个完全不同的项目，是否仍然有意义？有意义 → 合规；无意义 → 存在项目耦合，必须清除 |

---

### 第五步 — 定义与表现一致性

收集近期执行记录：

```bash
python cbim/memory/engine/cli.py query "<agent-name>" --top-k 15
python cbim/memory/engine/cli.py query "评审 <agent-name>" --top-k 5
```

| # | 因子 | 检查方法 |
|---|------|---------|
| 12 | 边界遵守 | 实际执行未越权（做了定义外的事）、未漏责（该做的没做） |
| 13 | 定义支撑表现 | 实际高频执行的操作，在 soul 或 skills 中有对应描述；无"不在定义内但一直在做"的隐性职责 |
| 14 | 表现支撑定义 | 定义的职责实际被使用；长期无任务的能力方向视为冗余，建议裂减或归档 |

---

## 输出：评审报告

```
评审时间：<YYYY-MM-DD>
Agent：<id>（<name>）

── 维度一：定义合理性 ──────────────────────────

基础格式（#1–3）：
  - [MUST/SUGGEST] [#因子编号] <问题> → <建议>

职责单一性（#4–5）：
  - [MUST/SUGGEST] [#因子编号] <问题> → <建议>

能力聚焦度（#6–8）：
  - [MUST/SUGGEST] [#因子编号] <问题> → 裂变 / 裁减 tools

内容纯洁性（#9–11）：
  - [MUST] [#因子编号] <具体耦合内容> → 移除，项目内容归入 .dna/

── 维度二：定义与表现一致性 ───────────────────

  - [MUST/SUGGEST/WARN] [#因子编号] <问题> → <建议>

── 汇总 ────────────────────────────────────────

MUST（必须修复）：<N> 条
SUGGEST（建议改进）：<N> 条
WARN（需确认）：<N> 条

处置结论：培训 / 裂变 / 归档 / 无需操作
后续行动：
  - 培训 → 执行 hr-training/SKILL.md
  - 裂变 → 执行 hr-agents/SKILL.md 裂变流程
  - 归档 → 执行 hr-agents/SKILL.md 归档流程
```

**严重级别定义**：
- `MUST` — 违反铁律（内容纯洁性耦合、soul 含项目内容）或严重格式缺失，必须修复
- `SUGGEST` — 可改进（职责偏宽、能力漂移、tools 冗余），建议处理
- `WARN` — 需人工判断（定义与表现不一致、长期闲置），上报助手确认
