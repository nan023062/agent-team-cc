# Skill: 记忆提炼（短期 → 中期）

**主 agent 专用。定期或按需触发。**

将短期 session 记录压缩为中期模式摘要，为 HR 能力治理和架构师业务治理提供原料。

---

## 触发时机

| 场景 | 说明 |
|------|------|
| 用户主动要求 | "提炼一下记忆" / "整理最近的 session" |
| 积累阈值 | `store/short/` 出现 ≥5 条未提炼 entry 时主动建议 |
| 治理前置 | HR 考核 / 架构师治理前，先运行本 skill 确保中期记忆最新 |

---

## 前置：填写信号

Stop hook 自动写入的 entry，信号行默认为空。**提炼前必须先补充信号**，否则无原料可提炼。

打开每个待处理的 `store/short/*.md`，在 `## 信号` 区填写观察到的内容：

```markdown
## 信号
- [x] 能力缺口：programmer: 并发写入场景下出现重复提交，未主动加锁
- [x] 优秀模式：architect: 先出 contract 再出 architecture，接口更稳定
- [ ] 知识更新候选：
```

**填写规范：**

| 信号类型 | 格式 | 写什么 |
|---------|------|--------|
| 能力缺口 | `agent-id: 描述` | agent 做不到、做错、需要用户纠正的事 |
| 优秀模式 | `agent-id: 描述` | 值得复用的方法、判断、工作习惯 |
| 知识更新候选 | `模块名: 描述` | 接口变更、设计决策、反复出现的业务逻辑 |

无信号则留空，不删行。填完后继续下面的步骤。

---

## Step 1 — 扫描短期 entry，收集填写过的信号

读取 `memory/store/short/` 下所有待提炼的 entry（已处理过的可跳过）：

```bash
ls memory/store/short/
```

对每个 entry，读取 `## 信号` 区，只收集已勾选的行（`- [x]`开头）。未勾选行忽略。

将收集到的信号按**维度**分组：

| 维度 | 来源信号类型 | 分组 key |
|------|------------|---------|
| **能力** | 能力缺口、优秀模式 | agent-id（如 `programmer`） |
| **业务** | 知识更新候选 | 模块名（如 `combat`） |

---

## Step 2 — 判断是否值得提炼

**不是所有信号都需要立刻写入中期。**

| 情形 | 处理方式 |
|------|---------|
| 同一 agent / 模块的信号在多条 entry 中重复出现 | **必须提炼**：重复出现说明是规律，不是偶然 |
| 单次出现，但描述了明确的缺口或决策 | **建议提炼**：判断是否值得记录为模式 |
| 单次偶发，上下文极为特殊 | 可暂不提炼，保留在短期即可 |
| 知识更新候选（无论次数） | **始终提炼**：这是架构师治理的直接输入 |

---

## Step 3 — 写入或更新中期 entry

中期文件路径规则：
- 能力维度：`memory/store/medium/capability-<agent-id>.md`
- 业务维度：`memory/store/medium/business-<module-name>.md`

**若文件已存在 → 更新；若不存在 → 新建。**

### 中期 entry 格式

```markdown
---
tier: medium
type: capability          # capability | business
keyword: programmer       # agent-id 或模块名
updated: YYYY-MM-DD
sources: 5                # 累计来源 entry 数量
---

## 摘要

（对所有信号的综合判断，用一段话概括这个 agent / 模块目前呈现的模式）

示例（能力维度）：
programmer 在并发写入场景下缺乏主动加锁意识，需要用户提示才会处理竞争条件。
单线程顺序任务表现稳定，擅长拆解步骤和调用工具链。

示例（业务维度）：
combat 模块的伤害计算接口在两次重构中均发生签名变更，现行签名为 calculate(actor, target, context)。
buff 叠加逻辑尚无 workflow，每次实现方式不一致，是架构师治理的优先候选。

## 信号记录

| 日期 | 来源 entry | 信号类型 | 内容 |
|------|-----------|---------|------|
| 2026-05-10 | 2026-05-10-main-xxx.md | 能力缺口 | 并发写入未加锁 |
| 2026-05-12 | 2026-05-12-main-yyy.md | 优秀模式 | 主动拆分子任务后效率提升 |

## 治理建议

- [ ] 触发 HR 考核（能力缺口 × N 次）
- [ ] 触发 HR 培训（优秀模式提炼为 skill）
- [ ] 触发架构师治理（知识更新候选 × N 个模块）
```

### 更新已有 entry 的规则

1. 在 `## 信号记录` 表格末尾追加新行
2. `sources` 计数 +新增 entry 数
3. `updated` 改为今天日期
4. 重新审视并改写 `## 摘要`（反映最新信号，而不是追加堆砌）
5. 根据新增信号更新 `## 治理建议` 中的勾选状态

---

## Step 4 — 清理已处理的短期 entry

保留最近 3 天（供 session 连续性使用），删除更早的：

```bash
.venv/bin/python -m memory.engine.cli cleanup --keep-days 3
```

---

## Step 5 — 汇报提炼结果

向用户输出本次提炼摘要：

```
## 记忆提炼摘要（{日期范围}，{N} 条 entry）

### 能力信号（{N} 条）
| agent | 类型 | 摘要 |
|-------|------|------|
| programmer | 缺口 × 2 | 并发场景加锁意识不足 |
| architect  | 模式 × 1 | 先 contract 后 architecture |

### 业务信号（{N} 条）
| 模块 | 类型 | 摘要 |
|------|------|------|
| combat | 知识更新 × 2 | 伤害接口签名变更，buff 逻辑待提炼 |

### 建议后续动作
- HR 考核 programmer（能力缺口重复出现）
- 架构师治理 combat 模块（接口变更 + workflow 缺失）
```

提炼完成后是否触发 HR 考核 / 架构师治理，由用户决定。
