# CBIM — Capability-Business Independence + Memory

> Claude Code의 컨텍스트 관리 프레임워크. 멀티 에이전트는 팀 시뮬레이션이 아니라 능력 축을 따라 컨텍스트를 격리하는 메커니즘입니다.

**CBIM** = **CBI**（Capability-Business Independence, 능력-비즈니스 독립성）+ **M**（Memory, 메모리 시스템）

## 해결하는 문제

가장 일반적인 Claude Code 워크플로우: **하나의 기본 에이전트 + 많은 CLAUDE.md 규칙 + 많은 스킬**.

이 패턴에는 시간이 지남에 따라 악화되는 구조적 결함이 있습니다: 대화 턴이 증가하면서 CLAUDE.md와 스킬 파일이 전체 컨텍스트에 로드되어 토큰이 폭발적으로 증가하고, LLM이 "중간에서 길을 잃고", 환각 발생률이 높아지며, 출력 품질이 저하되고, 수정이 컨텍스트를 더욱 오염시킵니다.

세션을 리셋하면 컨텍스트는 지워지지만 다른 문제가 발생합니다: 메모리 손실. 코드베이스를 다시 grep하고, 구조를 다시 이해하고, 에이전트에게 프로젝트 배경을 수동으로 다시 설명해야 합니다.

CBIM은 두 가지를 동시에 해결합니다:

| 문제 | CBIM 해결책 |
|------|------------|
| 턴 수와 함께 컨텍스트 팽창 | 멀티 에이전트 × 모듈 토폴로지 트리: 각 태스크에서 대상 에이전트 소울 + 태스크 서브트리 `.dna/`만 로드 |
| 세션 리셋 후 메모리 손실 | SessionStart 훅이 모듈 스냅샷 + 최근 메모리 자동 주입 — 비용 없이 복원 |

---

## 설계 철학

핵심 = **멀티 에이전트(능력 축) × 모듈 토폴로지 트리(비즈니스 축)**

- **능력 축**: 전문화된 에이전트 — 각 태스크에서 대상 에이전트의 소울만 로드, 불필요한 능력 컨텍스트 없음
- **비즈니스 축**: `.dna/`는 모듈 경계로 토폴로지 트리를 형성하고, 태스크의 서브트리만 로드, 불필요한 비즈니스 컨텍스트 없음
- **메모리**: 세션 간 축적된 원자료 — 세션 복원, 능력 거버넌스(HR이 distill → 스킬 → 소울), 비즈니스 거버넌스(아키텍트가 distill → `.dna/` 워크플로우)의 공통 소스

각 태스크의 컨텍스트 = 전문 에이전트 소울 × 태스크 서브트리 `.dna/` — 프로젝트 총 규모와 무관.  
컨텍스트 감소 → 환각 감소 → 오류 감소 → 수정 감소 → 단일 대형 에이전트보다 순 토큰 비용 절감.

## 실행 역할 (컨텍스트 격리 메커니즘)

CBIM은 전문화된 에이전트를 사용하여 능력 축을 따라 컨텍스트를 격리합니다 — 각 태스크에서 대상 에이전트의 소울만 로드하며 불필요한 능력 컨텍스트는 없습니다. 이것은 팀 시뮬레이션이 아니라 컨텍스트 제어입니다.

```
사용자
  ↓
어시스턴트 (CLAUDE.md — 유일한 외부 인터페이스, 태스크 라우팅)
  ├── 아키텍트   비즈니스 레이어 거버넌스: 프로젝트 지식 설계 및 유지(.dna/)
  ├── HR         능력 레이어 거버넌스: 워크 에이전트 라이프사이클 관리
  ├── 감사관     독립적 비판 리뷰 (적대적 관점, 읽기 전용)
  └── 워크 에이전트   구체적 태스크 실행 (HR이 필요에 따라 생성)
```

어시스턴트에게만 말하면 됩니다. 어시스턴트가 의도를 이해하고, 태스크를 분해하고, 적합한 에이전트에게 라우팅하고, 결과를 통합합니다.

## 빠른 시작

### 옵션 1: Claude Code를 통한 원라인 설치 (권장)

대상 프로젝트 디렉토리에서 Claude Code를 열고 이 메시지를 보내세요 — 에이전트가 모든 설치 단계를 자동으로 완료합니다:

```
Please fetch https://raw.githubusercontent.com/nan023062/cbim/master/INSTALL.md to get the CBIM installation SOP, then execute all steps starting after the first divider line to install in the current project.
```

### 옵션 2: 수동 설치

```bash
# 1. 대상 프로젝트의 cbim/ 디렉토리에 CBIM 클론
git clone https://github.com/nan023062/cbim.git cbim

# 2. 설치 프로그램 실행
python3 cbim/install.py        # macOS / Linux
# 또는 cbim/install.bat 더블클릭  # Windows

# 3. Claude Code 재시작
claude
```

## 디렉토리 구조 (배포 후)

```
your-project/
├── CLAUDE.md                      ← 어시스턴트 아이덴티티 (메인 세션)
├── .venv/                         ← Python 가상 환경 (gitignore)
│
├── .claude/
│   ├── settings.json              ← 권한 설정 + 훅 등록
│   └── agents/
│       ├── architect/             ← 아키텍트
│       ├── hr/                    ← HR
│       ├── auditor/               ← 감사관
│       └── programmer/            ← 프로그래머 (기본 워크 에이전트)
│
├── .dna/                          ← 프로젝트 지식 루트 모듈 (아키텍트가 생성)
│   ├── index.md
│   ├── module.json
│   ├── architecture.md
│   └── contract.md
│
└── cbim/                          ← 프레임워크 (여기에 git clone)
    ├── install.py                 ← 자동 설치 프로그램
    ├── install.bat                ← Windows 진입점
    ├── cc-template/               ← Claude Code 설치 템플릿
    ├── knowledge/                 ← 지식 엔진 (능력 + 비즈니스 CRUD)
    ├── memory/                    ← 메모리 엔진 (FileBackend)
    └── preview/                   ← 로컬 시각화 서버
```

## 2계층 거버넌스 · 2종류의 스킬

| 계층 | 관리자 | 대상 | 규칙 |
|------|--------|------|------|
| **능력 계층** | HR | `.claude/agents/`(소울) + `cbim/knowledge/skills/`(능력 스킬) | 소울/스킬에 프로젝트별 내용 미포함 |
| **비즈니스 계층** | 아키텍트 | 각 프로젝트의 `.dna/`(모듈 지식 + workflows/) | 지식 파일에 에이전트 사양 미참조 |

| 유형 | 저장 위치 | 특징 |
|------|---------|------|
| **능력 스킬** | `cbim/knowledge/skills/` | 에이전트 전용 능력; 이식 가능; HR 관리 |
| **비즈니스 스킬** | `.dna/workflows/` | 모듈 확정적 프로세스; 프로젝트 연결; 아키텍트 관리 |

## 메모리 시스템

메모리는 단순한 컨텍스트 복원이 아닌 3단계 증류 파이프라인입니다:

| 단계 | 경로 | 목적 |
|------|------|------|
| 단기 | `cbim/memory/store/short/` | 원시 세션 기록; 처리 후 `distilled` 태그, 3일 이상 보관 후 정리 |
| 중기 | `cbim/memory/store/medium/` | 압축된 패턴 요약; 지식 계층 승격 후 보관 |
| 지식 (핵심) | `cbim/knowledge/skills/` + `.dna/` | 결정화된 구조: 능력 → 스킬/소울, 비즈니스 → 워크플로우 |

SessionStart 훅은 세션 시작 시 자동으로 주입합니다: 프로젝트 지식 스냅샷(모듈 트리 + 에이전트 목록) + 이전 세션 복원 포인트 + 최근 메모리.
