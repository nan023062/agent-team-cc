# CBIM — Capability-Business Independence + Memory

> Claude Code のコンテキスト管理フレームワーク。マルチエージェントはチームシミュレーションではなく、能力軸でコンテキストを分離するメカニズムです。

**CBIM** = **CBI**（Capability-Business Independence、能力-業務独立性）+ **M**（Memory、記憶システム）

## 解決する問題

最も一般的な Claude Code の使い方：**1つのデフォルトエージェント + 大量の CLAUDE.md + 大量のスキル**。

このパターンには、時間とともに悪化する構造的な欠陥があります：会話のターンが増えるにつれて、CLAUDE.md とスキルファイルが全量コンテキストに読み込まれ、トークンが爆発的に増加し、LLM が「中間で迷子になり」、幻覚率が上昇し、出力品質が低下し、修正がさらにコンテキストを汚染します。

セッションをリセットするとコンテキストはクリアされますが、別の問題が発生します：記憶の喪失。コードベースを再 grep し、構造を再理解し、エージェントにプロジェクトの背景を手動で再説明する必要があります。

CBIM は両方を同時に解決します：

| 問題 | CBIM の解決策 |
|------|--------------|
| ターン数とともにコンテキストが膨張 | マルチエージェント × モジュールトポロジーツリー：各タスクで対象エージェントのソウル + タスクサブツリー `.dna/` のみを読み込む |
| セッションリセット後の記憶喪失 | SessionStart フック がモジュールスナップショット + 最近の記憶を自動注入 — ゼロコストで復元 |

## 設計哲学

コア = **マルチエージェント（能力軸）× モジュールトポロジーツリー（業務軸）**

- **能力軸**：専門化されたエージェント — 各タスクで対象エージェントのソウルのみを読み込み、余分な能力コンテキストなし
- **業務軸**：`.dna/` はモジュール境界でトポロジーツリーを形成し、タスクのサブツリーのみを読み込み、余分な業務コンテキストなし
- **記憶**（Memory）：セッション横断で蓄積された素材 — セッション復元、能力ガバナンス（HR が distill → スキル → ソウル）、業務ガバナンス（アーキテクトが distill → `.dna/` ワークフロー）の共通ソース

各タスクのコンテキスト = 専門エージェントのソウル × タスクサブツリーの `.dna/` — プロジェクトの総規模に依存しない。  
コンテキストが少ない → 幻覚が少ない → エラーが少ない → 修正が少ない → 単体大エージェントよりネットトークンコストが低い。

---

## 実行ロール（コンテキスト分離メカニズム）

CBIM は専門化されたエージェントを使用して能力軸でコンテキストを分離します — 各タスクで対象エージェントのソウルのみを読み込み、余分な能力コンテキストはありません。これはチームシミュレーションではなく、コンテキスト制御です。

```
ユーザー
  ↓
アシスタント（CLAUDE.md — 唯一の外部インターフェース、タスクルーティング）
  ├── アーキテクト   業務層ガバナンス：プロジェクト知識体系の設計・維持（.dna/）
  ├── HR             能力層ガバナンス：ワークエージェントのライフサイクル管理
  ├── 監査官         独立した批判的レビュー（対立的視点、読み取り専用）
  └── ワークエージェント   具体的なタスクを実行（HR が必要に応じて作成）
```

アシスタントとだけ話せばよい。アシスタントが意図を理解し、タスクを分解し、適切なエージェントにルーティングし、結果をまとめます。

## クイックスタート

### オプション 1：Claude Code への一行指示（推奨）

対象プロジェクトディレクトリで Claude Code を開き、このメッセージを送信してください — エージェントが全インストール手順を自動的に完了します：

```
Please fetch https://raw.githubusercontent.com/nan023062/cbim/master/INSTALL.md to get the CBIM installation SOP, then execute all steps starting after the first divider line to install in the current project.
```

### オプション 2：手動インストール

```bash
# 1. CBIM を対象プロジェクトの cbim/ ディレクトリにクローン
git clone https://github.com/nan023062/cbim.git cbim

# 2. インストーラーを実行
python3 cbim/install.py        # macOS / Linux
# または cbim/install.bat をダブルクリック  # Windows

# 3. Claude Code を再起動
claude
```

## インストール後の初回使用

Claude Code を再起動した後、送信：

> **"このプロジェクトのモジュール知識システムを初期化してください"**

アシスタントがアーキテクトに `.dna/` 知識システムの構築を指示します。その後、通常通り使用できます。

## 使い方

エージェントを指定せずに、アシスタントに直接伝えるだけです：

| やりたいこと | 伝え方 |
|-------------|--------|
| 知識システムの初期化 | このプロジェクトのモジュール知識システムを初期化してください |
| 機能モジュールの作成 | combat モジュールを作成してください |
| 機能の実装 | 現在の設計書に従ってログイン API を実装してください |
| 設計のレビュー | この変更をレビューしてください |
| 過去の決定の照会 | combat モジュールの決定履歴を教えてください |
| ワークエージェントの採用 | AI エンジニアエージェントを採用してください |

## 2層ガバナンス · 2種類のスキル

| 層 | 管理者 | 対象 | ルール |
|----|--------|------|--------|
| **能力層** | HR | `.claude/agents/`（ソウル）+ `cbim/knowledge/skills/`（能力スキル） | ソウル/スキルにプロジェクト固有の内容を含めない |
| **業務層** | アーキテクト | 各プロジェクトの `.dna/`（モジュール知識 + workflows/） | 知識ファイルにエージェント仕様を参照しない |

| 種類 | 保存場所 | 特徴 |
|------|---------|------|
| **能力スキル** | `cbim/knowledge/skills/` | エージェント固有の能力；ポータブル；HR が管理 |
| **業務スキル** | `.dna/workflows/` | モジュールの確定的プロセス；プロジェクトに紐付き；アーキテクトが管理 |

## 記憶システム

記憶は3段階の蒸留パイプラインであり、単なるコンテキスト復元ではありません：

| 段階 | パス | 目的 |
|------|------|------|
| 短期 | `cbim/memory/store/short/` | 生のセッション記録；処理後に `distilled` タグ付け、3日以上保持後クリーンアップ |
| 中期 | `cbim/memory/store/medium/` | 圧縮されたパターンサマリー；知識層に昇格後アーカイブ |
| 知識（コア） | `cbim/knowledge/skills/` + `.dna/` | 結晶化された構造：能力 → スキル/ソウル、業務 → ワークフロー |

SessionStart フックはセッション開始時に自動的に注入します：プロジェクト知識スナップショット（モジュールツリー + エージェントリスト）+ 前回セッション復元ポイント + 最近の記憶。
