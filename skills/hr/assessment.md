# Skill: 人才考核

## 触发条件

| 信号来源 | 说明 |
|---------|------|
| 任务完成情况 | 一批任务结束后，执行中出现频繁阻塞、未完成、重复犯错 |
| 用户指出不足 | 用户直接反馈 agent 表现问题，权重最高，立即触发 |
| 评审官持续打回 | auditor 多次标记同一 agent 的交付质量问题，说明系统性缺口 |

## 步骤

**Step 1 — 收集信号**

按触发来源读取材料：
- **任务完成情况** — 读取 `memory/<id>/` 下的信号记录（`memory/hr/agent-signals.md`）；梳理：完成了什么、哪里卡住、同类错误出现几次
- **用户反馈** — 记录用户的原话，作为最高优先级输入
- **评审官打回记录** — 读取 auditor 针对该 agent 的历史审查报告（若有），统计被标记的问题类型

同时读取被考核 agent 的 `.claude/agents/<id>.md`，确认其职责定位与工作规范。

**Step 2 — 判断问题性质**

区分两类问题：

| 类型 | 表现 | 说明 |
|------|------|------|
| 能力缺口 | 不会做、做不好，但方向对 | 培训可以修复 |
| 定位错误 | 反复接手不属于自己的任务，或核心职责理解有偏差 | 需重塑 soul/identity |

**Step 3 — 输出考核结论**

| 结论 | 标准 | 后续行动 |
|------|------|---------|
| 良好 | 整体符合预期，偶有小问题 | 仍建议培训升格，沉淀优秀模式 |
| 需培训 | 存在能力缺口，但定位正确 | 触发培训 skill，针对性填补缺口 |
| 需重塑 | soul/identity 设计有根本偏差，培训无法修复 | HR 重写 soul/identity，必要时重建 agent |

**Step 4 — 汇报秘书**

```
## 考核报告（{agent} · {日期}）
### 信号来源概览
### 问题性质判断（能力缺口 / 定位错误）
### 考核结论
### 建议后续行动
```

秘书决定是否触发培训或重塑流程。

---

## 重塑流程（结论为「需重塑」时执行）

重塑 ≠ 归档再重建。区别在于：soul/identity 设计有根本偏差，但 agent 的**存在价值不变**，只是"人设出了问题"。

### 判断：重塑 vs 归档重建

| 情况 | 处理 |
|------|------|
| 职责定位偏差，但能力方向正确 | 重塑（重写 soul/identity） |
| 职责与团队已有 agent 高度重叠 | 归档（用 recruitment skill 归档流程） |
| 职责已不再需要 | 归档 |
| 职责需要但 agent 完全不对路 | 归档 + 重新招募 |

### 重塑步骤

**Step R1 — 诊断根因**

对比 `.claude/agents/<id>.md` IDENTITY 节的职责定位 与 实际执行记录中的行为：
- 哪些职责被反复错误执行？
- soul 的哪些价值观/立场导致了偏差？
- 是 SOUL 问题（性格/立场跑偏）还是 IDENTITY 问题（职责边界不清）？

**Step R2 — 起草重塑方案**

针对诊断出的问题，起草修改内容：
- SOUL 偏差 → 重写对应章节（性格/立场/情感），保留正确部分
- IDENTITY 偏差 → 重写定位/职责/边界章节，保留技术规范等正确内容
- 可移植性检查：修改后内容仍须与项目无关

**Step R3 — 提交用户确认**

向秘书汇报：
- 诊断出的根因
- 修改前 vs 修改后对比
- 预期效果

用户确认后方可写入。

**Step R4 — 写入并通知**

- 更新 `.claude/agents/<id>.md`
- 在 `memory/<id>/` 新增重塑记录：日期、根因、变更摘要
- 通知秘书：重塑完成，下次 spawn 起生效
